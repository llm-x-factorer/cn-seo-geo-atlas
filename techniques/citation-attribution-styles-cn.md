# 中文 LLM 引用归因形态横向对照

> ⚠️ **状态：concept**（引用形态本身是**可观察事实**，但形态 → 用户回流 → 内容方商业价值的传导**几乎完全空白**——服务商话术里的"被引用 = 流量"在中文场景未被实证）
>
> 本卡和前 9 张技术卡的取向再一次不同：
> - 前 7 张是**内容供给侧**（怎么做内容）
> - 第 8 张（`prompt-level-seo-cn`）是**用户输入侧**
> - 第 9 张（`multimodal-cn`）是**内容形态侧**
> - 本卡是**LLM 输出侧**——LLM 怎么把"引用"呈现给用户，以及这种呈现对内容方的实际价值
>
> 本卡做四件事：
>
> 1. 拆开 6 大中文 LLM 引用形态的可观察特征：是否有上标编号 / 行内引用 / 卡片化 / 末尾列表 / 时间戳 / 缩略图 / 可点击链接
> 2. 区分"被引用"的两层价值：**信号价值**（用户看到品牌名）和**回流价值**（用户点击进站）——服务商话术常把这两层混为一谈
> 3. 横向对照中文 LLM vs Perplexity / ChatGPT Search 等英文场景基线
> 4. 反"被 AI 引用 = 等同 SEO 排名"话术
>
> ⚠️ **怀疑锚点 1**：**"被 LLM 引用"的商业价值远低于 SEO 时代"被搜索引擎收录"**——SEO 用户在结果列表里挑（CTR 2-5%）；GEO 用户接收一段合成文本（引用源 CTR < 1.3%，**这是 Google 数据，中文场景大概率更低**）。把 SEO 排名思维直接套到"被 AI 引用"是**两个量级的错觉**。
>
> ⚠️ **怀疑锚点 2**：引用形态本身不可优化——**这是 LLM 厂商的 UI 决定**，不是内容方能影响的。但**触发哪种引用形态**（如长文 vs 短答 / 决策类 vs 创意类）和**进入哪种信源池**与内容方的内容策略相关。
>
> ⚠️ **怀疑锚点 3**：[C-SEO Bench](https://openreview.net/forum?id=oTeixD3oZO) 在英文场景对各类"被引用次数"作为效应指标的反向证据值得参照——单纯的"被引用"是表层指标，**用户是否真消费、是否回流、是否产生品牌印象**才是内容方关心的最终因变量。

## 一、引用归因的 4 维分解

任何 LLM 答案中的引用都可以按 4 个独立维度刻画：

| 维度 | 取值 | 用户体验影响 |
|------|------|-------------|
| **位置** | 行内（"根据 X..."）/ 上标编号（[1]）/ 末尾列表 / 卡片块 | 上标 + 末尾列表是主流，卡片化最显著 |
| **可点击性** | 完全无链接 / 文字带链接 / 卡片可点击 / 可预览悬停 | 不可点击 → 不可回流（CTR = 0） |
| **元数据丰富度** | 仅 URL / URL + 标题 / URL + 标题 + 摘要 / + 缩略图 + 来源 logo | 越丰富 → 用户越愿意点击 |
| **可信度信号** | 来源平台 logo / 蓝 V 标识 / 时间戳 / 类型标签 | 直接影响用户对引用的"信任决策" |

### 1.1 6 大中文 LLM 引用形态横向对照

> ⚠️ 形态会随各厂 UI 改版**频繁变化**——本表为 2026-05 当前观察的快照，不保证未来稳定。

| LLM | 位置 | 可点击性 | 元数据丰富度 | 可信度信号 | 中文场景特殊处理 |
|-----|------|---------|--------------|------------|------------------|
| 豆包 | 上标编号 + 卡片化 | 卡片可点击 | URL + 标题 + 摘要 + 缩略图 | 来源 logo（抖音 / 头条 / 第三方） | 抖音视频源带视频缩略图 + 时间戳 |
| DeepSeek（联网模式） | 上标编号 + 末尾列表 | 链接可点击 | URL + 标题 | 域名展示 | 不绑定单一搜索引擎，引用源较"原生" |
| 文心 / 文心助手 | 行内引用 + 末尾列表 | 链接可点击 | URL + 标题 + 摘要 | 百度系特定标识 | 百度文库 / 百度健康 / 百度百科有专项卡片样式 |
| 千问 APP | 上标编号 + 链接列表 + 部分场景卡片 | 链接可点击 | URL + 标题 + 摘要 | 阿里系电商内容带商品卡 | 商品类 query 触发"AI 下单"卡片，与传统引用形态不同 |
| 元宝 | 上标编号 + 卡片化（公众号源单独样式） | 链接可点击 | URL + 标题 + 摘要 + 公众号头像 | **微信公众号蓝 V 直接展示** | 微信公众号源**在 APP 内可直接调起公众号阅读**，不跳出 |
| Kimi | 行内引用 + 末尾列表 | 链接可点击 | URL + 标题 | 域名展示 | URL 触发抓取场景下"原始 URL"高保真 |
| 百度搜索 AI 答案位 | 答案下方"参考资料"卡片 | 卡片可点击 | URL + 标题 + 缩略图 | 百度系来源标识 | 与 SERP 蓝链直接竞争用户注意力 |

### 1.2 与英文 LLM / Google AI Overviews 的对照

| 平台 | 位置 | 可点击性 | 元数据丰富度 | 中文 LLM 对应物 |
|------|------|---------|--------------|-----------------|
| Perplexity | 上标编号 + 末尾详细 + 卡片预览 | 强 | URL + 标题 + 摘要 + 来源 logo | 接近豆包 / 元宝形态 |
| ChatGPT Search | 类似 Perplexity（编号 + 详细） | 强 | URL + 标题 + 摘要 | 接近 DeepSeek / 千问形态 |
| Gemini | 简洁 / 部分场景给链接 | 中 | URL + 标题 | — |
| Google AI Overviews | 答案下方"展开来源" | 中 | URL + 标题 + 缩略图 | 类似百度搜索 AI 答案位 |
| Bing Copilot | 上标编号 + 末尾列表 | 强 | URL + 标题 + 摘要 | 类似 DeepSeek 联网模式 |

**关键观察**：

- 中文 LLM 卡片化最显著的是**豆包 / 元宝**——与生态优势直接相关（豆包整合抖音视频、元宝整合公众号文章）
- DeepSeek 引用形态最"中性"——与其"无生态"定位一致
- 文心 / 百度搜索 AI 是**百度生态内特殊样式 + 通用样式**双轨

## 二、被引用的两层价值——信号价值 vs 回流价值

服务商话术常笼统讲"被 LLM 引用"——但这个表述至少包含两层完全不同的价值：

### 2.1 信号价值（symbolic value）

- 用户**看到品牌名 / 域名**出现在 LLM 答案中
- 即使不点击，也产生"知道有这个品牌 / 信源"的认知
- 类似传统媒体广告的曝光维度
- **可观测面**：被引用次数 / 占比 / 在哪些 query 下出现

### 2.2 回流价值（traffic value）

- 用户**点击引用链接**进入站点
- 产生实际访问 / 购买 / 注册等行为
- **可观测面**：引用源 CTR / 点击进站后的转化率

### 2.3 两层价值的解耦

**关键事实**：信号价值和回流价值**可以独立存在**——

- 高信号 + 低回流：被多次引用但 CTR 接近 0（用户在答案中已得到信息，无动力点击）
- 低信号 + 高回流：很少被引用但被引用时 CTR 高（特定垂直行业 + 特定 query）
- 高信号 + 高回流：理想态，但需要内容形态 + 引用形态共同支持
- 低信号 + 低回流：被忽视

**服务商话术的混淆**：把"被引用次数"作为单一指标，等于把信号价值默认转换成回流价值——这是**没有依据的假设**。中欧白皮书引用的 Google 场景下"AI Overview 引用源 CTR 1-1.3%"恰恰提示**回流价值在 GEO 时代被显著压缩**。

### 2.4 中文场景的回流价值 = 多少？

- **公开数据**：**几乎完全空白**——没有任何中文 LLM 厂商公开过引用源 CTR
- **行业推测**：大概率**比 Google 场景的 1-1.3% 还低**，因为：
  - 中文 LLM 答案合成度更高（文本完整度高，用户更没必要点击）
  - 中文用户长期被"豆瓣 / 知乎 / 小红书内嵌答案"训练成"答案就在 APP 内"
  - 微信公众号引用在元宝场景**直接调起 APP 内阅读**，绕过传统 CTR 概念

## 三、引用形态触发的逻辑（推测）

> ⚠️ **本节大部分为推测**——LLM 厂商均未公开"何时触发卡片 / 何时触发行内引用"的规则。

| 触发因素（推测） | 卡片化引用更可能 | 行内 / 简洁引用更可能 |
|------------------|------------------|----------------------|
| Query 类型 | 决策 / 商品 / 视频 | 知识 / 创意 / 闲聊 |
| 信源类型 | 视频 / 商品 / 公众号 / 百科 | 普通网页 / 学术论文 |
| 答案长度 | 长答案 + 多源汇总 | 短答案 |
| 信源数量 | 多于 3 源 | 1-2 源 |
| 内容方控制度 | **几乎无**——决定权在 LLM 厂商 UI 设计 | 同左 |

**关键观察**：内容方对**触发哪种引用形态**的影响极弱——核心是 LLM 的 UI 设计决定。但**触发哪种 query 类型**（通过内容主题选择）部分可控。

## 四、待验证元假说

### 元假说 A：中文 LLM 的引用源 CTR 显著低于 Google AI Overviews 的 1-1.3%

- 表述：5 大中文 LLM 答案中"引用源被点击"的 CTR 中位数 < 1%（弱效应阈值之下）
- **驱动力**：答案合成度高 + 中文用户的"答案在 APP 内"训练 + 移动端为主
- **如何验证**：需要 LLM 厂商或大流量站点合作披露——本仓库能力之外，但可作为已有数据的求证目标
- **含义**：服务商话术"被 LLM 引用 = 流量入口"在中文场景大概率显著夸大

### 元假说 B：卡片化引用形态比纯文字引用的 CTR 高 3-5 倍

- 表述：在同一 LLM 上，卡片化展示的引用源（含缩略图 + 摘要）CTR 显著高于仅文字行内引用
- **如何验证**：需要 LLM 厂商内部数据；外部可设计实验——同一内容在豆包 / 元宝上分别触发卡片化 vs 行内的引用形态，比较站点的入站流量
- **副作用**：本假说成立则内容方应优先做"易触发卡片化"的内容形态——视频 / 商品 / 公众号

### 元假说 C：微信公众号源在元宝中的引用是中文场景**唯一接近"私域回流闭环"**的形态

- 表述：元宝 APP 内引用公众号文章时，可直接调起公众号阅读 + 关注按钮，绕过浏览器 CTR 测算——这种"软回流"是中文 LLM 场景独有的特殊路径
- **驱动力**：腾讯生态闭环（元宝 + 微信同主体）
- **如何验证**：观察元宝引用公众号源时的 UI 路径是否真的不跳出 APP；订阅转化率如何
- **关联**：[`platforms/yuanbao/citation-mechanism.md`](../platforms/yuanbao/citation-mechanism.md)

### 元假说 D：上标编号 vs 行内引用对用户回流的影响是次要变量

- 表述：在控制其他变量（可点击性 + 元数据丰富度）时，"上标编号 vs 行内引用"对用户回流的影响 < 5pp（弱效应）
- **驱动力**：用户决策的核心是"是否点击"，编号或行内是表面差异
- **含义**：服务商话术"上标编号引用更易触达用户"大概率夸大其词

### 元假说 E：被 LLM 引用的"信号价值"在长期看 > 单次"回流价值"——但仅限于已有品牌识别度的内容方

- 表述：对**已有品牌识别度**的内容方（用户能记住域名 / 品牌），被 LLM 多次引用的累积信号价值 > 单次点击回流；对**无品牌识别度**的内容方，被引用而无回流是**仅消耗算力的零价值事件**
- **副作用**：把"被引用"作为 GEO 单一目标对**新品牌 / 长尾内容方**是 ROI 误判——对它们而言"被引用"不构成商业闭环
- **关联**：与传统 SEO 时代"无品牌站点也能从蓝链 CTR 起家"形成结构性对照

## 五、实验设计建议

### 实验 1：中文 LLM 引用源 CTR 测量（验证元假说 A）

- **目标**：第一次量化中文 LLM 的引用源 CTR
- **设计**：
  - 内容方在自有站点埋 referrer 跟踪
  - 监测来自 doubao.com / chat.deepseek.com / yuanbao.tencent.com 等的入站流量
  - 同期记录被引用次数（人工或半自动通过 [`exp-doubao-vs-deepseek-paired`](../experiments/exp-doubao-vs-deepseek-paired.md) 工具栈采集）
  - CTR = 入站次数 / 引用次数
- **难点**：被引用次数难以精确统计；referrer 在移动 APP 端常缺失
- **优先级**：⭐⭐⭐ 最高——这是 GEO 商业价值的核心实证缺口

### 实验 2：内容形态触发的引用形态对照（验证元假说 B）

- **目标**：观察"什么内容触发卡片化引用"
- **设计**：
  - 同主题分别用（纯文本博客 / B 站视频 / 公众号文章 / 商品页）发布
  - 对豆包 / 千问 / 元宝 / 文心 / DeepSeek 跑相同 query
  - 记录引用形态（卡片 / 行内 / 末尾）
- **优先级**：⭐⭐ 中

### 实验 3：元宝公众号引用的回流路径观察（验证元假说 C）

- **目标**：核实"元宝引用公众号是否绕过传统 CTR 测算"
- **设计**：观察 + 描述性研究——记录元宝 APP 内点击公众号引用的实际 UI 路径
- **优先级**：⭐⭐ 中（结论可能有特殊操作意义）

### 不建议做的实验

- **"上标编号 vs 行内引用"对照**：内容方无法控制 LLM 的展示形态，无可操作变量
- **"加多个 schema 类型是否提升卡片化引用"**：与 [`structured-data-cn-practice.md`](./structured-data-cn-practice.md) 元假说 B 同源，预期失败
- **"在内容里写'请点击查看更多'是否提升 CTR"**：LLM 大概率会过滤这类话术

## 六、与 GEO 服务商话术的对照

国内常见话术：

- "被 AI 引用 = AI 时代的 SEO 排名"
- "我们让你被豆包 / 文心 / 元宝引用，AI 流量不愁"
- "卡片化引用 = 黄金广告位"
- "被引用 N 次 / 月套餐"

**本卡片判定**：

| 话术 | 判定 | 理由 |
|------|------|------|
| "被 AI 引用 = SEO 排名" | ❌ 量级错觉 | 怀疑锚点 1；引用源 CTR < 蓝链 CTR 一个量级 |
| "AI 流量不愁" | ⚠️ 取决于品牌识别度 | 元假说 E；对无品牌内容方"被引用而无回流"是零价值 |
| "卡片化引用 = 黄金广告位" | ⚠️ 信号价值高，回流价值不一定 | 卡片化提升 CTR（元假说 B）但仍远低于传统 SERP CTR |
| "被引用 N 次 / 月套餐" | ❌ 单一指标误导 | 把信号价值默认换算成商业价值 |
| "做内容尽量进生态卡片化触发条件" | ✅ 操作合理 | 视频 / 公众号 / 商品类 query 卡片化触发概率高 |
| "已有品牌的话被引用是品牌强化" | ✅ 与元假说 E 一致 | 信号价值对已知品牌可累积 |

**结论**：服务商话术里**与"信号价值 + 已知品牌"对齐**的部分（卡片化触发 + 品牌强化）相对靠谱；**把"被引用次数"作为商业价值单一指标**的部分大概率夸大其词。

## 七、与本仓库其他卡片的关系

- 与 [`multimodal-cn.md`](./multimodal-cn.md)：多模态信源被引用时的卡片化形态（视频缩略图 / 商品卡）是本卡讨论的具体形态
- 与 [`prompt-level-seo-cn.md`](./prompt-level-seo-cn.md)：用户问法决定 query 类型，query 类型决定引用形态——两卡是因果链上下游
