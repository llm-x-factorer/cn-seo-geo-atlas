# 多模态信源在中文 LLM 中的采纳现状

> ⚠️ **状态：concept**（中文 LLM 多模态能力差异极大但**多模态信源 → LLM 引用率**的因果链几乎完全空白）
>
> 本卡补全前 8 张技术卡的一个共同盲区——它们**默认内容是文本**。但中文场景的内容生态高度多模态化：小红书图文 / 抖音视频 / B 站视频 / 微信公众号配图 / 文档站 PDF。本卡处理这一维度。
>
> 本卡做四件事：
>
> 1. 拆开"多模态"——图片 / 视频 / 音频 / PDF 不是同一回事，各有不同的**信号传导路径**
> 2. 横向对照 6 大中文 LLM 的多模态**输入能力**（用户能传什么）vs **召回能力**（RAG / 训练时怎么处理外部多模态信源）——这俩常被混淆
> 3. 拆"内容方做多模态优化"的具体抓手：图片 alt / OCR / 视频字幕 / 音频转录 / PDF 文本层——分别评估对 LLM 引用率的影响
> 4. 反"做图片 alt / 上传 4K 视频 / 加详细字幕提升 AI 引用"话术
>
> ⚠️ **怀疑锚点 1**："多模态 LLM 输入能力强"≠"对外部多模态信源召回能力强"。豆包能识图（用户上传图给豆包看）和豆包在 RAG 召回时会专门处理图片信源**是两个完全不同的能力**——前者已知（公开），后者**完全未公开**。
>
> ⚠️ **怀疑锚点 2**：多数 LLM 在训练 / RAG 时并不"真看图"——而是**预先用 OCR / ASR / 字幕提取**把多模态转为文本再走文本管道。如果这一假设成立，则**内容方做"图片 alt 优化"的边际效应大概率接近 0**——因为 LLM 走的是 OCR 自己做的文本提取，alt 标签可能根本不在它的视野里。
>
> ⚠️ **怀疑锚点 3**：[C-SEO Bench](https://openreview.net/forum?id=oTeixD3oZO) 在英文场景对各类"内容信号优化"的反向证据同样适用此处——"加 alt 文本 / 加字幕"的边际效应**大概率显著小于服务商宣称值**。

## 一、多模态≠一种东西：4 类信源的差异

| 类型 | 中文典型载体 | 文本提取常见路径 | 在 LLM 训练 / RAG 中的可见性 |
|------|--------------|------------------|---------------------------|
| **图片** | 小红书图文配图 / 公众号配图 / 海报 / 截图 | OCR + alt 属性 + 周围文本 | 中—弱（OCR 准确率 + 召回不公开） |
| **视频** | 抖音 / B 站 / 视频号 / YouTube 中文 | ASR 字幕（平台自动生成）+ 标题 + 描述 + 弹幕 | 中（视频站点字幕大量被爬） |
| **音频** | 小宇宙播客 / 喜马拉雅 / B 站音频 | ASR 转录 + 节目页文本 | 弱—中（中文 ASR 质量参差） |
| **PDF / 文档** | 学术论文 / 政府文件 / 行业白皮书 | PDF 文本层（如有）+ OCR 兜底 | 中—强（文本层完整时质量高） |

**关键观察**：4 类信源的**信号传导路径完全不同**——把它们笼统讲成"多模态优化"会丢失差异。本卡之后所有讨论按这 4 类分别展开。

## 二、中文 LLM 的"输入能力"vs"召回能力"——两套独立维度

服务商话术常拿"豆包能识图"作为"做多模态信号"的依据——这是**类别错误**。

| 能力维度 | 含义 | 谁可观测 |
|----------|------|----------|
| **多模态输入能力** | 用户上传图片 / 视频 / 音频 / PDF，LLM 能否理解并回答 | **公开**——各家 LLM 都展示能力 |
| **多模态召回能力** | LLM 在 RAG 检索 / 训练数据爬取时，对外部多模态信源（如小红书图、抖音视频）的处理方式与权重 | **几乎完全未公开** |

这两个能力**没有必然耦合**——一个 LLM 能在用户上传时识图（输入侧），完全可能在外部 RAG 时只用文本（召回侧）。

### 2.1 各 LLM 输入能力对照（公开层面）

| LLM | 图片输入 | 视频输入 | 音频输入 | PDF 输入 | 备注 |
|-----|---------|---------|----------|----------|------|
| 豆包 | ✅ 原生支持 | ✅（部分场景） | ✅ | ✅ | 字节系多模态线整合 |
| 千问 | ✅ Qwen-VL 系 | ✅ Qwen-VL 系 | ✅ Qwen-Audio | ✅ | 阿里多模态线最完整开源 |
| 文心 | ✅ | ✅ | 部分 | ✅ | 文心一格 / 文心视频 |
| 元宝 | ✅（依赖混元 + DeepSeek 多模型路由） | ✅ | 部分 | ✅ | 多模型路由架构 |
| DeepSeek | ⚠️ 服务侧默认文本 | ❌ | ❌ | ⚠️ | DeepSeek-VL 仅开源，chat 默认文本 |
| Kimi | ⚠️ 部分 | ❌ | ❌ | ✅ **强**（长文档解析是核心卖点） | 长文本 + 文档为主，图片视频弱 |

### 2.2 召回能力（推测，**完全未公开**）

| LLM | 推测的召回路径 | 推测的多模态信源依赖度 |
|-----|---------------|---------------------|
| 豆包 | 字节系生态内（抖音视频字幕 / 头条配图） | 高（生态优势） |
| 千问 | 阿里电商生态（淘宝商品图 / 短视频）+ 通用爬取 | 中—高 |
| 文心 | 百度生态（百度图片 / 好看视频 / 百度文库 PDF） | 中 |
| 元宝 | 微信生态（公众号图文 / 视频号）+ 多模型路由 | 中 |
| DeepSeek | 通用爬取，多模态信源大概率被丢弃或转文本 | 弱 |
| Kimi | 用户上传 + 通用爬取；外部多模态依赖弱 | 弱 |

**核心推测**：与文本信源相同的模式——**生态内 LLM 对生态内多模态信源召回强**，跨生态弱。

## 三、内容方能做的"多模态优化"逐项评估

### 3.1 图片 alt 属性

- **是什么**：HTML `<img alt="...">` 属性，可机读的图片描述
- **传统 SEO 价值**：Google / 百度图片搜索均使用，影响图片搜索可见性 + 无障碍访问
- **中文 LLM 价值**：**未公开**——多数 LLM 在 RAG 召回时大概率走 OCR + 周围文本，而**不专门读 alt**
- **建议**：仍然写（无障碍访问 + 图片搜索仍然有效），但**不要预期 GEO 提升**

### 3.2 图片 OCR 文本

- **是什么**：图片中的文字（海报标语 / 截图代码 / 思维导图等）
- **中文场景特殊性**：小红书图文是"图片中文字"为主体的形态——文字在图里而不在图旁
- **LLM 处理**：依赖各 LLM 自己的 OCR 能力 + 是否在爬取 / RAG 时执行
- **建议**：**重要文字不要只放在图里**——即使为了视觉做图片，把核心文本也以纯文本形式放在文档里

### 3.3 视频字幕

- **是什么**：内嵌字幕 / SRT 字幕 / 平台自动生成 ASR 字幕
- **中文场景特殊性**：抖音 / 视频号 / B 站均提供平台自动 ASR——内容方甚至不需要自己做
- **LLM 处理**：抖音 / B 站字幕大概率被字节系 / B 站站内 LLM 召回（字节系 → 豆包路径）；通用 LLM 是否爬取这些字幕未公开
- **建议**：**长视频要有字幕**——双重价值（用户体验 + 部分 LLM 召回路径）；自动 ASR 已经够用，不必专门做

### 3.4 音频转录

- **是什么**：播客 / 音频内容的文字稿
- **中文场景特殊性**：小宇宙 / 喜马拉雅 / 网易云等多有平台 ASR；公众号文章里手动附音频转录较少见
- **LLM 处理**：极弱——音频原文件几乎不被通用 LLM 爬取
- **建议**：**有播客的同时附文字稿**（在节目主页或公众号文章中），文字稿才是 LLM 可见的形态

### 3.5 PDF 文本层

- **是什么**：PDF 内嵌的可选中文本 vs 扫描图 PDF 需 OCR
- **LLM 处理**：文本层完整的 PDF 在通用爬取中相对友好；扫描 PDF 依赖 OCR 质量
- **建议**：**PDF 必须含完整文本层**——这是基础工程，不是优化技巧

### 3.6 多模态结构化数据（VideoObject / ImageObject）

- **是什么**：schema.org 中针对视频 / 图片的专用类型
- **中文场景**：与 [`structured-data-cn-practice.md`](./structured-data-cn-practice.md) 同一维度——百度等部分支持，中文 LLM 解析未公开
- **建议**：与一般 schema 一致——做也无妨但不预期 GEO 提升

## 四、中文场景独有的多模态信源生态

### 4.1 小红书图文（"图片中的文字"为主体）

- 笔记主体往往是**图片中文字**（文案设计在图里）+ 图下文本（话题标签 + 简短描述）
- **关键 LLM 路径**：抓取小红书时**必须 OCR 才能拿到主体文本**——这对通用爬取是高门槛
- **推测**：豆包 / 千问可能通过特殊路径处理小红书图片；其他 LLM 大概率丢失主体内容
- **关联**：[`platforms/xiaohongshu/overview.md`](../platforms/xiaohongshu/overview.md) 的"图文双轨"特征

### 4.2 抖音 / 视频号 / B 站视频

- 视频站点通常对外提供：标题 + 描述 + 评论（可爬）+ 字幕（部分可爬）
- **B 站特殊性**：弹幕也是文本信号，且数量级大，但被通用 LLM 爬取的程度未公开
- **AI 总结功能**：B 站 / 视频号近年推出"AI 视频总结"——这本质是用户侧 LLM 调用，不直接成为内容方的多模态信号

### 4.3 微信公众号配图

- 公众号编辑器**支持图文混排**——配图通常带 alt 但不可机读控制（编辑器封装）
- **OCR 路径**：通用爬取公众号文章时，配图是否被 OCR 取决于爬虫是否做这层
- **元宝路径**：腾讯生态内，元宝可能对公众号配图有特殊处理（推测）

### 4.4 政府 / 学术 PDF

- 政府公文 / 学术论文是中文 PDF 信源的核心
- 文本层完整度通常较好（早年扫描件较多，近年逐步改善）
- **百度文库 / 知网（CNKI）/ 万方等**是 PDF 信源的中文聚合中转
- 中文 LLM 是否专门处理 PDF 信源未公开

## 五、待验证元假说

### 元假说 A：中文 LLM 在外部多模态信源召回上的能力分层

- 表述：豆包 / 千问 / 文心（生态内多模态信源丰富）≥ 元宝 / DeepSeek（生态弱 / 通用爬取） > Kimi（最弱）
- **驱动力**：生态多模态资源 + 厂商投入意愿
- **如何验证**：受控对照——同主题在多模态形态（图片为主 / 视频为主 / PDF 为主）下，5 大 LLM 引用率差异
- **关联**：[`eeat-cn.md`](./eeat-cn.md) 元假说 B（生态作为权威性代理）的多模态侧延伸

### 元假说 B：内容方做"图片 alt 优化"的边际效应接近 0

- 表述：在 alt 完整 vs 无 alt 的同质内容下，5 大中文 LLM 引用率差异 < 5pp（弱效应阈值）
- **驱动力**：LLM 在 RAG 召回时多走 OCR + 周围文本，alt 不在主路径
- **如何验证**：A/B 对照——同主题"图片含 alt 完整 vs alt 缺失"
- **关联**：[`structured-data-cn-practice.md`](./structured-data-cn-practice.md) 元假说 B（schema 边际效应接近 0）的姊妹假说

### 元假说 C：视频字幕（特别是平台自动 ASR）是中文场景被 LLM 召回最多的多模态信号

- 表述：在所有多模态信号里，**视频平台自动生成的 ASR 字幕**因体量大 + 文本化程度高 + 站点级聚合，是中文 LLM 实际召回最多的多模态来源
- **含义**：**视频内容方什么都不用做**（不用 alt 不用结构化数据），只要在抖音 / B 站 / 视频号上发布即可被生态内 LLM 召回字幕——这与"做 alt"的低效形成对照
- **如何验证**：实际跑视频内容 vs 图文内容在豆包 / 元宝引用率差异

### 元假说 D：图片中的文字（OCR 文本）对 LLM 引用率的影响 > 图片 alt 属性

- 表述：把核心文字写在图里（小红书风格）后，LLM 实际通过 OCR 提取的文本对引用率的贡献 > 同等文本作为 alt 的贡献
- **驱动力**：OCR 走主路径文本管道，alt 走元数据路径
- **副作用**：本假说反直觉——传统 Web 标准建议**避免把文字放图里**（无障碍 + SEO 都偏好可选中文本）；但中文场景 LLM 的实际行为可能反过来奖励"图里有字"

### 元假说 E：通用 LLM（DeepSeek / Kimi）对中文多模态信源的召回率显著低于生态内 LLM

- 表述：在外部图片 / 视频信源命中率上，DeepSeek / Kimi 显著低于豆包 / 千问 / 元宝 / 文心
- **含义**：服务商话术"做多模态信号覆盖所有 LLM"在 DeepSeek / Kimi 上大概率失效
- **副作用**：以多模态内容为主的内容方在 DeepSeek / Kimi 上**结构性弱势**，需要在站外做文本对齐版本

## 六、实验设计建议

### 实验 1：图片 alt 完整 vs 缺失 A/B（验证元假说 B）

- **目标**：直接证伪 / 验证"做 alt 提升 LLM 引用率"话术
- **设计**：
  - 自变量：2 档（完整 alt + structured data ImageObject vs 完全无 alt）
  - 因变量：5 大中文 LLM 引用率
  - 控制：图片内容、配文、域名、发布时间
- **样本**：每档 ≥ 30 篇 × 30 天
- **优先级**：⭐⭐⭐ 最高

### 实验 2：图文 vs 视频 vs 纯文本 在生态内 vs 通用 LLM 的引用率（验证元假说 A + C + E）

- **目标**：量化生态内 LLM 对生态内多模态信源的优势
- **设计**：
  - 同主题三档（纯文本博客 / 小红书图文笔记 / B 站视频含字幕）
  - 5 大 LLM 引用率对照
- **样本**：每档 ≥ 20 主题 × 5 LLM × 5 重复
- **优先级**：⭐⭐⭐ 最高（直接量化"内容形态选择"的 GEO 收益）

### 实验 3：图里有字 vs 图外有字（验证元假说 D）

- **目标**：检验 OCR 路径 vs 文本路径的相对效应
- **设计**：
  - A 组：核心文字写在图里（小红书风格），周围文本简短
  - B 组：核心文字写在图外（传统博客风格），图片为辅助
- **优先级**：⭐⭐ 中（结论可能有反直觉的实操指导价值）

### 不建议做的实验

- **"上传 4K 视频是否提升 AI 引用率"**：清晰度与 LLM 召回无相关性（LLM 处理的是字幕 / 描述，不是像素）
- **"做 VideoObject schema 提升 AI 引用率"**：与 [`structured-data-cn-practice.md`](./structured-data-cn-practice.md) 元假说 B 同源，预期失败
- **多模态信源所有形态全交叉单实验**：变量过多无法归因

## 七、与 GEO 服务商话术的对照

国内常见话术：

- "做图片 alt 优化提升 AI 引用率"
- "上传高清视频 + 完整字幕，被 AI 优先收录"
- "多模态结构化数据是 AI 时代标配"
- "我们提供视频转录 + 字幕优化服务"

**本卡片判定**：

| 话术 | 判定 | 理由 |
|------|------|------|
| "图片 alt 提升 AI 引用率" | ❌ 大概率无效 | 元假说 B；alt 不在 LLM 主路径 |
| "上传高清视频 + 完整字幕" | ⚠️ 半对 | 字幕有效（元假说 C），但"高清"与 LLM 召回无关 |
| "VideoObject / ImageObject schema 标配" | ❌ 与 schema 卡同源结论 | [`structured-data-cn-practice.md`](./structured-data-cn-practice.md) 元假说 B |
| "视频转录 + 字幕优化" | ⚠️ 部分有效 | 抖音 / B 站平台 ASR 已经够用；专门做转录的边际收益有限 |
| "把核心文字也放在图里" | ✅ 与元假说 D 一致（反直觉） | 中文 LLM 走 OCR 路径，图里文字反而进入主路径 |
| "在抖音 / B 站发视频会被豆包 / 元宝引用" | ✅ 与元假说 C 一致 | 字幕 + 生态闭环 |

**结论**：服务商话术里**与"内容形态选择"对齐**的部分（视频在生态内、把核心字放在图里）相对靠谱；**与"传统 SEO 元数据迁移"对齐**的部分（alt / schema）大概率无效。
