# 中文知识图谱与百科生态横向对照

> ⚠️ **状态：concept**（百科生态结构事实清晰，但百科 → SEO 特型卡触发 → LLM 引用率传导链未实证）
>
> 本卡做四件事：
>
> 1. 把"知识图谱"和"百科"**区分清楚**——它们在中文场景被混用，但是两个层级的概念
> 2. 横向对照 6+ 大中文百科生态的所属、规则、采编门槛、与所属生态搜索 / LLM 的耦合度
> 3. 拆"百科收录 → SEO 可见性 → LLM 引用率"的因果链；尤其要把**维基中文**这个"被墙但被训练"的特殊案例单列
> 4. 反"代写百科必中 AI 引用"服务商话术
>
> ⚠️ **怀疑锚点 1**：百科在中文搜索特型卡（百度知识图谱卡 / 抖音知识答案 / 微信搜一搜聚合卡）里的占位是**接近事实**的——多年 SEO 实践共识。但"百科收录 → 中文 LLM 引用率提升"的因果链中**只有第一段（百度百科 → 文心）有较强的耦合证据**，其余推测居多。
>
> ⚠️ **怀疑锚点 2**：维基百科中文是个**特殊案例**——它在中国大陆受访问限制，但**很可能是所有中文 LLM 训练数据的底层之一**（Common Crawl 等公开数据集均含）。这意味着"做维基中文条目"的 GEO 路径和"做百度百科"完全不同——前者通过训练数据进入 LLM，后者通过 RAG 检索召回进入。
>
> ⚠️ **怀疑锚点 3**：[C-SEO Bench](https://openreview.net/forum?id=oTeixD3oZO) 在英文场景对各类"权威信源植入"的反向证据同样适用此处——"代写百度百科"在中文 GEO 上的实际效应量级**大概率小于服务商宣称值**。

## 一、知识图谱 vs 百科——不是同一概念

### 1.1 知识图谱（Knowledge Graph）

- **是什么**：一种**结构化知识表示**——以"实体—关系—实体"三元组为基本单位（"姚明—身高—226cm""姚明—配偶—叶莉"）
- **谁在做**：Google Knowledge Graph（2012）/ 维基数据 Wikidata / 百度知识图谱 / 阿里电商 KG / 腾讯医典 KG 等
- **形态**：可以是图数据库（Neo4j 类）、可以是 RDF 三元组、也可以是结构化卡片
- **用途**：搜索引擎的 SERP 知识面板、问答系统的事实型问答、推荐系统的实体关联

### 1.2 百科（Encyclopedia）

- **是什么**：一种**文章式知识载体**——以条目为基本单位，每条目包含定义 + 详情 + 来源 + 关联条目
- **形态**：HTML 页面（含富文本 + 表格 + 图片）
- **本质**：百科是**知识图谱的来源之一**——百科条目可以被解析、抽取三元组，进而构成知识图谱

### 1.3 两者在中文搜索 / LLM 中的角色

| 维度 | 知识图谱 | 百科 |
|------|---------|------|
| 数据形态 | 结构化三元组 / 图 | 自然语言长文 |
| 在 SERP 的呈现 | 知识面板（百度的特型聚合卡） | 普通蓝链 + 部分进入特型卡 |
| 在 LLM 的角色 | RAG 召回时作为事实型补充（需结构化解析） | 训练数据 + RAG 召回的"长文本上下文" |
| 维护主体 | 搜索引擎 / 平台自有 | 用户协作（多数）/ 平台编辑 |
| 中文场景采纳 | 百度知识图谱（早期高调，后期淡化）/ 阿里 / 腾讯医典等垂类 | **百度百科 / 抖音百科 / 维基中文 / 快懂百科** 等多生态并存 |

**关键观察**：本卡的横向对照**主要在百科层级**——因为知识图谱在中文场景的公开维度有限，且 LLM 时代百科作为"长文本训练 / RAG 召回源"的角色已经远比"知识图谱三元组源"更核心。

## 二、中文 6+ 大百科生态横向对照

### 2.1 概览

| 百科 | 所属 / 整合后归属 | 域名 | 规模 | 编辑门槛 | 与所属生态搜索 / LLM 的耦合 |
|------|-------------------|------|------|----------|----------------------------|
| **百度百科** | 百度 | baike.baidu.com | 极大（中文最大综合百科） | 账号等级 + 编辑通过率 + 引用规则 | **强**——百度搜索知识图谱卡 + 文心引用 |
| **维基百科中文** | 维基媒体基金会 | zh.wikipedia.org | 大 | 全球编辑社区 + 严格引用规则 | **特殊**——大陆访问受限但**训练数据耦合极强** |
| **抖音百科** | 字节（原互动百科 hudong.com 2020 并入） | baike.com | 中—大 | 用户编辑 + 字节系账号体系 | 与字节系搜索 / 豆包潜在耦合 |
| **快懂百科** | 360（原 360 百科改名） | baike.so.com | 中 | 用户编辑 + 360 账号 | 与 360 搜索耦合 |
| **搜狗百科** | 腾讯（搜狗被腾讯整合） | baike.sogou.com | 中 | 用户编辑 + 腾讯系账号 | 部分进入微信搜一搜 |
| **萌娘百科** | 独立维基（ACG 垂类） | zh.moegirl.org.cn | 中（垂类） | ACG 社区编辑 | 训练数据耦合（ACG 类 query 上） |
| **MBA 智库百科** | 智库 / 经管垂类 | wiki.mbalib.com | 中（垂类） | 用户编辑 | 经管类 query 训练数据 |
| **维基数据 Wikidata** | 维基媒体基金会 | wikidata.org | 极大（结构化三元组） | 全球编辑 | 知识图谱层而非百科层；中文 LLM 是否复用未公开 |

### 2.2 编辑门槛与"代写"灰产

| 百科 | 公开门槛 | 灰产现状（业内观察） |
|------|----------|---------------------|
| 百度百科 | V4+ 账号 + 引用央媒 / 权威源 + 多次驳回是常态 | 代写 / 代修服务多年存在，价格随条目类型差异大 |
| 维基中文 | 全球社区，规则最严，要求第三方独立可靠来源 | 代写难度极高，社区很快回滚 |
| 抖音百科 | 较低（用户编辑）+ 字节系账号 | 代写常见但稳定性弱（条目易被审核或下线） |
| 快懂 / 搜狗 | 较低 | 代写常见 |

**关键观察**：编辑门槛与**条目稳定性**成正比——百度百科 / 维基中文条目一旦稳定，长期可见性强；快懂 / 搜狗 / 抖音百科条目下线 / 改写更频繁。**"门槛低 = 容易做 = 也容易没"**。

## 三、百科 → 中文搜索引擎特型卡（接近事实的部分）

| 搜索引擎 | 主导百科信源 | 在 SERP 的呈现 |
|----------|--------------|----------------|
| 百度 | **百度百科** | 知识图谱聚合卡，强占位（详见 [`platforms/baidu/seo-signals-and-ai-overview.md`](../platforms/baidu/seo-signals-and-ai-overview.md) 第三节） |
| 神马 | 多源混合（无强主导） | 弱知识卡，公开度低 |
| 360 | **快懂百科**（自家） | 360 知识卡 |
| 搜狗 / 微信搜一搜 | **搜狗百科**（自家整合） | 微信搜一搜聚合卡偶尔展示 |
| 必应中文 | **维基中文** | 知识面板（与全球必应一致） |
| 抖音站内搜索 | **抖音百科** | 抖音知识答案卡 |

**核心模式**：**自家百科 → 自家搜索特型卡** 在中文场景是各家通行做法。这与 [`eeat-cn.md`](./eeat-cn.md) 元假说 B（自家生态作为权威性代理）同源——百科是这一模式的具体载体之一。

## 四、百科 → 中文 LLM 引用率（拆解 + 推测）

### 4.1 三条不同的传导路径

| 路径 | 描述 | 实证状态 |
|------|------|----------|
| **路径 1：训练数据** | 百科条目被纳入 LLM 训练数据 → 模型记忆 → 直接生成包含该知识的答案（无 RAG） | 维基中文路径接近事实；百度百科存疑（爬取困难度更高） |
| **路径 2：RAG 检索召回** | 百科条目被 RAG 实时检索 → 用作生成上下文 → 答案中可能引用 | 百度百科 → 文心耦合度高；其他配对未公开 |
| **路径 3：知识图谱抽取** | 百科条目被抽取为三元组 → 进入 KG → LLM 通过 KG 接口查询 | 中文 LLM 是否使用 KG 接口未公开 |

**关键观察**：路径 1（训练）和路径 2（RAG）是两类不同机制——前者**是模型自身记忆**，与发布时机无关；后者**实时检索**，对发布时机敏感。

### 4.2 各百科 → 各 LLM 的耦合度（推测）

| 百科 \ LLM | 文心 | 豆包 | DeepSeek | 元宝 | 千问 | Kimi |
|-----------|------|------|----------|------|------|------|
| 百度百科 | **强**（路径 2） | 弱—中（外站爬取） | 弱（通用爬取） | 弱（外站） | 弱 | 弱 |
| 维基中文 | 中（路径 1，所有都有） | 中（路径 1） | 中（路径 1） | 中（路径 1） | 中（路径 1） | 中（路径 1） |
| 抖音百科 | 弱 | **中—强**（生态内） | 弱 | 弱 | 弱 | 弱 |
| 快懂百科 | 弱 | 弱 | 弱 | 弱 | 弱 | 弱 |
| 搜狗百科 | 弱 | 弱 | 弱 | **中**（微信搜一搜路径） | 弱 | 弱 |
| 萌娘 / MBA 智库等垂类 | 弱 | 弱 | 弱 | 弱 | 弱 | 弱 |

**强对应路径**：

- 百度百科 → 文心（路径 2，RAG 实时召回）
- 抖音百科 → 豆包（生态内，多路径）
- 维基中文 → 全部 LLM（路径 1，训练数据）

其余配对**均为通用爬取**——大概率丢失百科作为"权威信源"的语义信号，被当作普通网页处理。

### 4.3 维基中文的特殊地位

维基中文在中文 GEO 中扮演**事实标准**角色，但有三个反直觉特性：

1. **大陆访问受限 ≠ 训练数据缺失**——维基中文 dump 是公开数据集，全球 LLM 厂商都能下载并训练
2. **维基条目稳定性极强**——一旦写入，长期被引用；但**编辑壁垒也极强**，社区会快速回滚利益相关编辑
3. **维基的"中立性"原则反向阻断 GEO**——维基要求第三方独立可靠来源，**禁止自我推广**——任何带商业利益的条目编辑大概率会被回滚

**结论**：维基中文是中文 GEO 的"看得见但够不着"的高地——它是事实标准，但**几乎无法被 GEO 服务化**。这是它和百度百科的本质区别。

## 五、待验证元假说

### 元假说 A：维基中文是所有中文 LLM 的训练数据底层之一（接近事实）

- 表述：5 大中文 LLM（豆包 / DeepSeek / 文心 / 千问 / 元宝 / Kimi）训练时**全部使用了维基中文**（来自 Common Crawl 或 Wikimedia 公开 dump）
- **驱动力**：维基中文是公开高质量中文长文本数据集，几乎不可能不用
- **如何验证**：fingerprint 注入实验——在维基中文条目里植入独特词（合规风险，社区极速回滚），观察是否在 LLM 输出中出现；或反向设计：用 LLM 复述特定罕见维基条目内容，看是否包含该条目的特异性细节
- **风险**：验证实验有合规 + 社区伦理风险，建议改用"罕见但已存在的条目"做被动观察

### 元假说 B：百度百科是中文场景"做百科"GEO 路径的唯一**生态闭环**

- 表述：百度百科 → 百度搜索知识图谱卡 → 文心引用——这一路径在 5 大中文 LLM 中**只有文心**有强耦合；其他 LLM 几乎不复用百度百科
- **含义**：服务商话术"代写百度百科保 AI 引用"在文心维度部分成立，**对其他 4 家 LLM 大概率失效**
- **如何验证**：5 平台 LLM 对照——同主题在"有百度百科条目 vs 无百度百科条目"的引用率差
- **关联**：[`eeat-cn.md`](./eeat-cn.md) 元假说 B（文心是唯一复用底层搜索 E-E-A-T 信号的中文 LLM）的同一现象

### 元假说 C：抖音百科 → 豆包 是字节系闭环

- 表述：在字节系搜索（抖音搜索 / 头条搜索）中，抖音百科条目优先；在豆包对应类 query 上，抖音百科条目命中率显著高于其他百科
- **如何验证**：豆包 vs 其他 LLM 对照"有抖音百科条目"的内容引用率差异
- **关联**：[`platforms/doubao/citation-mechanism.md`](../platforms/doubao/citation-mechanism.md)

### 元假说 D：百科 → SEO 特型卡的耦合 ≠ 百科 → LLM 引用的耦合

- 表述：百度百科条目在百度搜索 SERP 上的特型卡占位（强）**远大于**该条目在中文 LLM 答案中的引用率（中—弱）
- **含义**：服务商把"百度搜索特型卡占位"案例外推到"AI 引用率提升"是**跨变量混淆**——这是一种与 [`structured-data-cn-practice.md`](./structured-data-cn-practice.md) 第四节"跨变量失效"同源的错误
- **如何验证**：双因变量同时测——同条目 SERP 特型卡触发率 + 文心 / 豆包引用率

### 元假说 E：维基中文条目的稳定性使其成为"GEO 不可优化"的硬基线

- 表述：维基中文已有的条目对中文 LLM 输出有结构性影响（路径 1 训练）；但**新增 / 修改维基中文条目的 GEO 收益接近 0**——因为：(1) 编辑被回滚的概率极高；(2) 即使留存，也不能进入已训练好的模型（除非下次训练），且不能用于 RAG（大陆受访问限制）
- **含义**：维基中文是"看的见的天花板"，是事实参照而非操作目标
- **副作用**：把维基中文当成 GEO 操作目标的服务话术（"我们能写维基条目"）在 ROI 维度大概率不成立

## 六、实验设计建议

### 实验 1：百度百科 vs 抖音百科 vs 快懂 vs 搜狗 在 5 大 LLM 上的引用率对照（核心实验）

- **目标**：验证元假说 B + C（生态内闭环）
- **设计**：
  - 自变量：4 档（仅百度百科 / 仅抖音百科 / 仅快懂 / 仅搜狗百科）
  - 因变量：5 大中文 LLM 引用率（文心 / 豆包 / DeepSeek / 元宝 / 千问）
  - 控制：query 主题、时段
- **样本**：每档 ≥ 10 个主题 × 5 平台 × 5 重复 = 1000 次采集
- **难点**：需创建 4 类百科条目并保持稳定（涉及代写灰产边界，建议选**没有商业利益的中性主题**）
- **优先级**：⭐⭐⭐ 最高
- **复用**：[`exp-doubao-vs-deepseek-paired`](../experiments/exp-doubao-vs-deepseek-paired.md) 工具栈扩展即可

### 实验 2：维基中文已有条目 vs 无维基条目的 LLM 输出差异（验证元假说 A）

- **目标**：观察维基中文作为训练数据的实际传导
- **设计**：选择"维基中文已有完整条目"和"维基中文无条目"两类同等冷僻度的实体名词，让 LLM 复述，看输出的特异性细节是否吻合维基条目
- **优势**：被动观察，不需要新建条目
- **风险**：需控制冷僻度变量（条目存在与否本身相关于该实体的网络存在度）
- **优先级**：⭐⭐ 中

### 实验 3：百度百科 SERP 特型卡 vs LLM 引用率双因变量观察（验证元假说 D）

- **目标**：拆开"SERP 特型卡触发"和"LLM 引用率"两个因变量
- **设计**：跟踪一组百度百科条目，同时记录其在 SERP 上的特型卡占位率和在文心 / 豆包答案中的引用率，看两者相关性
- **优先级**：⭐⭐ 中（适合做基础数据回写）

### 不建议做的实验

- **"维基中文条目 fingerprint 注入"**：合规 + 社区伦理风险，且回滚速度太快——反向被动观察更稳
- **"代写百度百科服务购买对比"**：进入灰产边界，仓库不收
- **多百科 + 多 LLM + 多 query 全交叉单实验**：变量过多，建议分阶段

## 七、与 GEO 服务商话术的对照

国内常见话术：

- "做百度百科必中 AI 引用"
- "我们提供百科代写服务，覆盖百度 / 搜狗 / 360 / 抖音四大百科"
- "百科是 AI 时代的 SEO 硬通货"
- "维基中文条目是顶级权威，我们能搞定"

**本卡片判定**：

| 话术 | 判定 | 理由 |
|------|------|------|
| "百度百科必中 AI 引用" | ⚠️ 仅文心维度部分成立 | 元假说 B；其他 4 家 LLM 几乎不复用百度百科 |
| "百科代写覆盖四大百科" | ⚠️ 操作可行，效应分散 | 各百科 → 各 LLM 是分裂的耦合，覆盖不等于乘加 |
| "百科是 AI 时代 SEO 硬通货" | ⚠️ "SEO 硬通货"接近事实，"AI 时代"夸大 | SERP 特型卡的耦合强；LLM 引用的耦合弱（元假说 D） |
| "维基中文条目是顶级权威，能搞定" | ❌ 与维基社区规则冲突 | 元假说 E；商业利益相关编辑被回滚概率极高 |
| "做抖音百科有助于豆包引用" | ✅ 与元假说 C 一致 | 字节系生态闭环 |

**结论**：服务商话术里**与生态闭环对齐**的部分（百度百科→文心、抖音百科→豆包）相对靠谱；**跨生态外推**的部分（百度百科→豆包 / DeepSeek）大概率失效。
