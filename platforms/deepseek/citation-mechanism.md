# DeepSeek 引用机制（公开信息整理 + 待验证）

> ⚠️ **可信度声明**：与豆包卡片同样的限制——**没有官方文档背书，没有作者亲自实测**。所有"信源偏好"都来自第三方 GEO 服务商分析，应当被视为待验证的**假说**。
>
> DeepSeek 的特殊性：开源 + 大量第三方接入，使得"DeepSeek 的引用机制"可以指至少 3 个不同的对象（见下文"研究边界"）。

## 研究边界（与豆包不同的关键问题）

在写"DeepSeek 引用机制"之前必须先界定**研究的是哪个对象**：

| 研究对象 | 数据来源 | GEO 优化的可控点 |
|----------|----------|------------------|
| **A. DeepSeek 官方 APP / chat.deepseek.com** | 模型 + 官方 RAG 配置 | 类似豆包，目标是被官方检索 + 引用 |
| **B. 通过 DeepSeek API 的第三方应用**（元宝、问小白、纳米 AI、知乎直答…） | 模型 + **每个接入方自己的 RAG** | **每个集成方独立优化**，跨产品不可迁移 |
| **C. 自部署 DeepSeek 模型** | 仅模型训练数据，**无联网检索** | 只能影响"未来训练数据"层 |

本卡片**主要讨论 A**。B 和 C 留作未解问题。

## 假说一：技术社区与新闻媒体偏好

> 来源：CSDN 博客《AI 搜索推荐机制深度拆解》（2026），泽森科技团队报告

声称的引用源权重排序（**未独立验证**）：

| 信源类型 | 声称权重 | 备注 |
|----------|----------|------|
| **CSDN 博客** | 高 | 技术 query 下的核心引用源 |
| 搜狐号、网易号 | 中高 | 通用资讯 |
| 百科站点（百度百科、维基中文、抖音百科等） | 中高 | 知识类 query 的稳定信源 |
| 知乎技术问答 | 中 | 社区内容 |
| GitHub 技术文档 | 中 | 编程 / 技术 query |

**为什么这个假说在结构上合理**（不是背书，是说为什么值得验证）：

- DeepSeek 用户画像偏技术 / 编程 / 推理，query 分布也偏技术
- DeepSeek 自己**没有内部内容生态**，所以必然用通用 / 公开内容源
- 技术社区（CSDN、知乎、GitHub）天然是中文技术 query 的高质量信源池
- 模型本身在训练时见过大量代码 + 技术问答数据，结构亲和

**为什么不能直接采信**：

- "高 / 中高" 这种定性描述没给方法学
- 服务商可能有"代发 CSDN 博客"业务，存在动机偏差
- 不同 query 类型下分布完全不同（编程 query 和"如何投资"的引用结构不可比）

**待办**：`experiments/deepseek-citation-source-distribution.md`——明确 query 集（编程 / 技术 / 通用知识 / 决策类各 N 条）+ 多次重复 + 公开数据。

## 假说二：联网检索是否启用 / 何时启用

DeepSeek 官方 APP 的联网搜索行为有几个**待澄清**的事实点：

1. 是否所有 query 都触发联网检索？还是有"何时联网"的策略？
2. 用户是否能在 UI 中显式切换"联网搜索"开关？
3. 联网检索的引擎是 DeepSeek 自建还是接第三方（如 Bing 中文 / 必应 / 自研爬虫）？
4. 推理模型（R1 / R 系列）和通用模型（V 系列）的联网行为是否相同？

**这些是机制层面的事实问题**，应该可以通过仔细观察 UI / 抓包 / 跨 query 对照来确认。但本卡片不预设结论。

## 假说三：技术内容长效引用，无明确时效窗口

> 来源：CSDN 博客分析

声称：

- DeepSeek 对内容"无明确时效期限"，技术内容长期有效
- 与豆包"1-2 周内权重最高"形成对比

**结构性合理性**：

- 技术问题的答案随时间变化慢（"如何排序数组"答案稳定）
- 资讯类 query 在 DeepSeek 上不是核心使用场景

**但需要细分**：

- 编程 / 数学 / 推理 query：长效合理
- 时事 / 政策 / 财经 query：仍需时效
- 这条假说**仅在 query 类型条件下成立**

## 假说四：结构化偏好（与豆包共通）

声称偏好的内容结构：

- 代码块（```...```）
- 架构图、流程图（Mermaid 图、ASCII 图）
- 参数表
- 步骤化解释（"原理 → 实现 → 边界"）

**这条相对可信**，因为：

- 与所有 LLM 的检索增强 chunking 行为一致
- 技术内容的结构化偏好在所有平台都成立，不是 DeepSeek 特有

## 假说五：与豆包的核心差异（最有意思的对照）

如果其他假说都成立，那么**豆包和 DeepSeek 的引用结构差异**应该是这样：

| 维度 | 豆包 | DeepSeek |
|------|------|----------|
| **生态偏好** | 字节系内部内容显著加权 | 无明显生态偏好（公开技术社区为主） |
| **时效要求** | 1-2 周内最优 | query 类型决定，技术类长效 |
| **核心信源** | 头条、抖音、抖音百科、快懂百科 | CSDN、搜狐号、网易号、百科 |
| **品牌植入风格** | 信息流、生活化、自然提及 | 技术实践者分享、原理分析 |
| **多模态** | 强（视频转写、抖音视频内容） | 偏文本、代码、表格 |
| **决策 / 消费 query 表现** | 强（接入抖音商城） | 弱（无电商生态） |
| **技术 query 表现** | 一般 | 强 |

**这张对照表的研究价值**：

- 如果中文 GEO 的"信源偏好"在豆包和 DeepSeek 上**显著不同**：那 GEO 优化策略**必须按平台分流**，"通用 GEO 策略"概念需要被打破
- 如果**反而趋同**：那"生态偏好"在中文 LLM 中可能被高估（GEO 服务商的话术含水分）

这是仓库可以贡献的**第一个高价值实证发现**——值得优先做配对实验验证。

## 反假说 / 怀疑信号

调研中观察到的不一致信号：

1. **DeepSeek 模型本身不等于 DeepSeek APP**：很多对"DeepSeek 引用 CSDN"的观察来自第三方接入方（不是 chat.deepseek.com），这些观察其实是**接入方 RAG 配置**的体现
2. **R1 vs V 模型的差异**：用户感知中"R1 推理时引用更多""V 模型直接回答"——但这可能只是不同 UI 设置
3. **2026-03 V4 发布后的行为变化**：很多老的 GEO 服务商分析基于 V3 / R1 时代行为，V4 之后可能不再适用

## 关于"接入方分歧"问题（B 类研究边界）

通过 DeepSeek API 接入的产品：

| 产品 | 接入方式 | 它自己的 RAG / 信源 |
|------|----------|-----------------------|
| 腾讯元宝（部分场景） | API 调用 R1 | 接微信公众号、视频号生态 |
| 问小白 | 集成 DeepSeek + 其他模型 | 自建检索 |
| 纳米 AI 搜索（360） | 集成多模型 | 360 生态 + 第三方 |
| 知乎直答 | 集成 DeepSeek + 自有模型 | **知乎自有内容** |
| Cursor / Cline / Coding Agent | 调用 API 做编程任务 | 通常无 RAG（基于本地代码） |

**对 GEO 实践的含义**：

- 你想"被 DeepSeek 引用"是个**模糊目标**——必须先回答"通过哪个产品被引用"
- 优化策略**不可跨接入方复用**：让你的内容在元宝 R1 模式下被引用，与让你的内容在知乎直答中被引用，是两件事
- "DeepSeek API 不是 GEO 的优化对象"——优化对象始终是接入方的 RAG 配置

## 在做实验前能下的结论

**可以下的（强）**：

- DeepSeek 月活第三、人均使用次数仅次于豆包（一手数据）
- DeepSeek 没有自有内容生态——这是**结构性事实**
- DeepSeek 通过 API 被大量第三方应用接入——这是公开事实
- 因此"DeepSeek 引用机制"必须先界定研究对象（A/B/C 三类）

**不能下的**：

- "DeepSeek 偏好 CSDN" 的程度量化
- 联网检索的具体策略
- R1 vs V 在引用行为上的差异
- 与豆包的具体差异程度（需要配对实验）
