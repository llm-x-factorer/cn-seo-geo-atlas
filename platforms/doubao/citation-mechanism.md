# 豆包引用机制（公开信息整理 + 待验证）

> ⚠️ **可信度声明**：本卡片整理自第三方 GEO 服务商和自媒体的公开分析，**没有官方文档背书，没有作者亲自实测**。所有数字（占比、时效窗口、权重排序）都视为**假说**，应当通过 [`../../experiments/`](../../experiments/) 中的实验复核。
>
> 字节官方未公开豆包的内容采纳 / 信源遴选规则，所以这里的"机制"全部来自**外部观察 + 推断**，请按假说强度阅读。

## 假说一：内部生态优先、外部信源补充

> 来源：CSDN 博客《AI 搜索推荐机制深度拆解》(2026)、GEO 服务商榜单文章（2026）

声称的引用源权重排序（**未独立验证**）：

| 信源类型 | 声称占比 / 权重 | 备注 |
|----------|------------------|------|
| 今日头条深度文章 | 60%+ | 数字来自 GEO 服务商口径 |
| 抖音内容（视频图文 / 评论 / 描述） | 高 | 推测含转写文本 |
| 抖音百科 / 快懂百科 | 高 | 字节系结构化知识库 |
| 知乎专业问答 | ~20% | 外部信源里相对靠前 |
| 西瓜视频 | 中 | 视频转写 |
| 其他外部资讯 / 自媒体 | 补充 | 网易号、搜狐号等 |

**为什么这个假说在结构上合理**（不是为它背书，而是说明为什么值得验证）：

- 豆包大模型的训练 / 微调数据天然倾向字节系内容（数据获取成本低、合规可控）
- 检索增强（RAG）链路上，字节内部内容的索引完整度、新鲜度、API 调用成本都最优
- 抖音百科 / 快懂百科是字节自营的结构化知识基础设施

**为什么不能直接采信第三方数字**：

- "60%+" 出自 GEO 服务商，他们有动机放大字节内容的重要性以销售自己的服务
- 不同 query 类型下分布会完全不同（"如何投资"和"豆包是什么"的引用结构没可比性）
- 没有给出测试方法（query 集、时间、统计口径）

**待办**：设计 `experiments/doubao-citation-source-distribution.md`，公开 query 集 + 抓取脚本 + 原始数据。

## 假说二：APP 端访问与 API 端访问的信源结构显著不同

> 来源：知乎专栏 2025-12 文章（[链接](https://zhuanlan.zhihu.com/p/2016929994631292520)）

声称的差异：

- **用户侧（APP / Web）**：高频引用今日头条、抖音、快懂百科等字节系自有 / 独家内容源
- **API 侧（火山引擎）**：信源结构更"通用"，字节系内容占比下降

**结构性解释**（推测）：

- APP 端服务于消费场景，背后的检索增强可能更依赖"内容运营 + 商业链路"
- API 端服务于开发者 / 企业，可能使用更通用的检索增强配置以满足合规与可移植性
- 这意味着**B 端开发者基于豆包 API 跑出来的"豆包行为"不能直接外推到 C 端用户体验**

**这条假说如果成立，对 GEO 实践的影响**：

- 用 API 做"豆包可见性监测"会**系统性低估**字节内部内容的真实权重
- 反过来：用 APP 做监测要解决自动化抓取问题（防爬、登录态、风控）

**待办**：`experiments/doubao-app-vs-api-divergence.md`——同时刻、同 query 双端跑，对照引用源。

## 假说三：时效性窗口偏短

> 来源：GEO 服务商文章

声称：

- 1-2 周内的内容权重最高
- 超过 1 个月引用率显著下降
- 实时数据（当月信息）引用概率提升 40%+

**为什么需要怀疑这个数字**：

- 没说明是哪类 query（"今天发生了什么"和"如何减肥"的时效需求天差地别）
- "实时数据引用概率提升 40%" 没有基线
- 服务商有动机把"时效性"做成"必须持续付费的理由"

**这条假说如果成立的实操含义**：

- 内容生产侧需要持续更新，而不是一篇定终身
- 监测侧需要把"近 7 天 / 近 30 天"作为基础时间切片

**待办**：`experiments/doubao-recency-window.md`——同一 query 在不同时间点观察答案，看引用源是否更替。

## 假说四：结构化偏好

> 来源：CSDN 博客、GEO 服务商榜单

声称豆包偏好的内容结构：

- FAQ 问答（Q：…… A：……）
- 列表对比（"三种方法"、"五个步骤"）
- 解释型短段落（200-400 字）
- 大段散文论述被去优先级

**这条相对可信**，因为：

- 与所有主流 LLM 的检索增强 chunking 行为一致（chunk 大小通常 200-500 token）
- 结构化内容更容易被切片、嵌入向量、命中检索
- 不只豆包，DeepSeek / 千问 / 文心 / Kimi 都有类似偏好

**实操含义**（普适，不限豆包）：

- 内容首段直接给定义 / 核心结论
- 用 H2 / H3 切分语义段
- 数据用表格、对比用列表
- 长文章拆成多个独立可检索的小块

但具体 chunking 策略、阈值、对中文标点的处理是黑箱。

## 假说五：E-E-A-T 信号在中文 GEO 场景的对应物

> 来源：艾瑞 2026 GEO 报告、GEO 服务商话术

声称的"权威信源"金字塔：

| 层级 | 声称 |
|------|------|
| 顶层 | 央媒（人民网、新华网、央视网……） |
| 高层 | 地方官媒、政府门户 |
| 中层 | 综合门户（新浪、网易、搜狐、腾讯新闻） |
| 中层 | 行业垂直媒体（36 氪、虎嗅、丁香园等） |
| 中下 | 自媒体（微信公众号、知乎、小红书、B 站） |
| 字节生态独立轨道 | 头条、抖音、快懂百科 |

**强烈警告**：

- "央媒 AI 采信率 92%" 这种数字几乎全部来自 GEO 服务商的销售材料
- "签央媒发稿就能让豆包引用你" **是商业话术，不是机制证据**
- 真实情况大概率是：央媒 / 政府背景内容在**某些 YMYL 类 query**（医疗、金融、政策）下被加权，但**不是所有 query 都这样**
- 服务商常见做法：用一个测试良好的 query 演示效果，避免谈失败 case

**这条假说要被验证为"机制"**（而非"商业现象"），需要：

- 控制变量地对比同主题、同信息量、不同信源的内容被引用率
- 区分"被检索到"和"被作为主要引用源"
- 跨 query 类型测试

## 已知的"反假说"信号

调研中也观察到一些**与上面假说不一致**的现象，需要进一步研究：

1. **GEO 服务商口径互相矛盾**：不同服务商给出的"权重排序"差异显著，没有形成一致的事实
2. **二手报告滞后**：很多 2026 年初的报告还在用"豆包月活 1.16 亿"的旧数字（实际 2026Q1 已 3.45 亿），其分析结论的时效性可疑
3. **"发央媒就有效"的反例**：用户社区有零星观察称央媒发稿后未观察到豆包引用变化，但这些观察也不严谨

## 在做实验前能下的结论

**可以下的（强）**：

- 豆包月活第一、用户使用频次远超同行（QuestMobile 一手数据）
- 字节生态闭环对其 GEO 行为有结构性影响（生态结构事实）
- API 端与 APP 端不能默认等同（多个独立来源指向）
- 结构化内容偏好是 LLM 通用偏好，非豆包特有

**不能下的**：

- 任何具体的占比数字（60%、20%、92%……）
- "签央媒就能上豆包"这类因果链条
- 时效性窗口的具体边界
- 跨 query 类型的稳定权重排序
