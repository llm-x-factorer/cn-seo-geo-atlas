# 元宝引用机制（公开信息整理 + 待验证）

> ⚠️ **可信度声明**：与豆包 / DeepSeek / 文心 / 千问 卡片同样的限制——**没有官方文档背书，没有作者亲自实测**。所有"信源偏好"都来自第三方 GEO 服务商分析和一线观察，应当被视为**假说**。
>
> 元宝的特殊性：**多模型路由 + 多入口 + 微信生态加持**——研究复杂度是仓库内最高的。

## 研究边界

元宝的研究边界比文心、千问还复杂：

| 研究对象 | 模型 | 数据源 | GEO 优化的可控点 |
|----------|------|--------|------------------|
| **A. 元宝 APP / yuanbao.tencent.com** | 多模型路由（混元 + R1 + 其他） | 公众号 + 通用 | 优化微信公众号文章 |
| **B. 微信内嵌 AI 搜** | 混元（推测） | **微信生态封闭**（公众号 + 视频号 + 小程序） | **微信公众号 + 视频号优化** |
| **C. QQ 浏览器 QBot** | 多模型 | 网页 + 浏览历史 + 智能体调度 | 网页 SEO + 智能体优化 |
| **D. 腾讯文档 AI** | 混元 | 文档内容 + 通用 | 文档结构化 |
| **E. 腾讯会议 AI** | 混元 | 会议转写 + 通用 | （非典型 GEO 场景） |
| **F. 企业微信 AI** | 混元 | 企业内部数据 | （企业内部场景） |
| **G. 腾讯云大模型 API** | 混元 | 接入方 RAG | 完全由接入方决定 |

本卡片**主要讨论 A 和 B**（最高频用户场景）。

## 假说一：微信公众号生态优先（最强护城河）

> 来源：一线观察（X / 推特）、GEO 服务商榜单文章

声称的引用源权重排序（**未独立验证**）：

| 信源类型 | 声称权重 | 备注 |
|----------|----------|------|
| **微信公众号文章** | **核心**（36 亿篇文章历史池） | 元宝最强加权信源 |
| **视频号内容** | 高（视频字幕 / 描述） | 微信生态多模态信源 |
| **腾讯文档** | 高（结构化文档） | B 端场景主导 |
| **小程序数据** | 中（结构化业务数据） | 特定 query |
| **通用 web 内容** | 中 | 补充 |
| **DeepSeek R1 路由场景** | （继承 R1 的信源偏好） | 复杂推理 query |

**为什么这个假说在结构上合理**：

- 微信公众号是**中文最大的私域内容池**——36 亿篇文章 + 长期更新 + 高质量长文
- 微信生态是**封闭的**——其他模型抓不全，所以元宝在公众号信源上有**结构性垄断优势**
- 与豆包"字节系优先"、文心"百度系优先"假说同构——**反向印证"中文大厂 LLM 普遍偏好自家生态"的元假说**
- 一线观察明确指出"微信公众号是进入元宝的最直接通道"

**这条假说强度高于其他平台同假说**——因为微信生态的封闭性使竞争对手几乎无法跟进。

**为什么仍需验证**：

- "36 亿篇文章" 是历史池规模，**实际被 RAG 检索的比例未知**
- 不同 query 类型下公众号占比可能波动
- 多模型路由下，部分 query 可能完全不走公众号信源
- 微信内嵌 AI 搜（B 类入口）vs 元宝 APP（A 类入口）的权重可能不同

**待办**：`exp-yuanbao-wechat-mp-bias`、`exp-yuanbao-citation-source-distribution`。

## 假说二：多模型路由策略（独特问题）

元宝的**独特复杂性**——同一个 query 可能被路由到不同模型：

| query 类型 | 推测路由 | 信源偏好 |
|------------|----------|----------|
| 简单事实 / 闲聊 | 混元 | 微信生态为主 |
| 复杂推理 / 数学 / 代码 | DeepSeek R1 | 继承 R1 的偏好（CSDN 等） |
| 多模态 / 视频 | 混元（视频处理能力较强） | 视频号 + 抖音公开内容 |
| 业务 / 工作场景 | 混元 + 文档 RAG | 腾讯文档 + 公众号 |

**对 GEO 实践的含义**：

- "在元宝上的引用"实际上是"在某个被路由的模型 + RAG 配置下的引用"
- 多模型混合 → **优化策略需要按 query 类型分流**
- 复杂推理类 query 可能完全脱离微信生态加成（走 R1 + 通用 RAG）

**这条假说如果成立**：

- 元宝 GEO 比其他平台更"分裂"——同时需要做微信公众号 + DeepSeek 信源（CSDN 等）
- 监测难度更高——需要识别每次回答用的是哪个模型

**待办**：`exp-yuanbao-multi-model-routing`——通过对照 query 观察路由策略。

## 假说三：微信内嵌 AI 搜的"封闭信源"特殊性

微信内嵌 AI 搜（B 类入口）可能**完全只用微信生态内信源**：

| 信源 | 是否在微信 AI 搜中可用 |
|------|------------------------|
| 微信公众号文章 | **是**（核心） |
| 视频号内容 | **是** |
| 小程序数据 | **是**（部分） |
| 朋友圈 | 否（隐私） |
| 微信群聊 | 否（隐私） |
| **百度搜索结果** | **否**（外部） |
| **抖音 / 头条 / 小红书** | **否**（外部） |
| 通用 web 内容 | 不确定（待实测） |

**如果"完全封闭"成立**：

- 微信内嵌 AI 搜是中文 GEO 中**最封闭的入口**——优化策略**只有发微信公众号 / 视频号一条路**
- 与百度搜索内嵌 AI（半开放）、豆包 APP（半开放）形成极端对比
- 对品牌方意味着：**重视微信公众号是必选项**，不只为了私域，更为了 AI 入口

**这条假说如果不成立**（微信内嵌 AI 搜也用部分外部信源）：

- 那研究复杂度会显著提高
- 但优化路径仍然有微信生态优先的强偏向

**待办**：`exp-yuanbao-app-vs-wechat-embedded`。

## 假说四：与五大厂的对照表（最终对照）

整合五个 GEO 平台的关键引用机制对照——**仓库的最终对照表**：

| 维度 | 豆包 | DeepSeek | 文心 | 千问 | **元宝** |
|------|------|----------|------|------|----------|
| **核心引用源** | 头条、抖音、抖音百科 | CSDN、搜狐号、网易号 | 百家号、百度百科、百度知道 | 网易号、搜狐号 | **微信公众号、视频号、腾讯文档** |
| **生态偏好强度** | 强（字节） | 几乎无 | **最强（百度）** | 弱（与电商生态绑定不深） | **最强（微信生态封闭性）** |
| **生态封闭程度** | 部分封闭（抖音视频） | 无生态 | 部分封闭（百度索引） | 部分（淘宝商品数据） | **最封闭（微信公众号 / 视频号）** |
| **时效偏好** | 1-2 周 | query 类型决定 | 3 个月 | 6 个月 | 待验证（推测中等） |
| **结构偏好** | FAQ / 列表 / 信息流 | 代码块 / 架构图 | 首段定义 / 数据 / 总结句 | 表格 / 列表 / 模块化 | **长文 / 公众号风格 / 深度** |
| **品牌植入风格** | 信息流自然提及 | 技术实践者 | 权威解读者 | 第三方报道口吻 | **深度长文 / 私域权威账号** |
| **多模型架构** | 单一 | 单一 | 单一 | 单一 | **多模型路由（混元 + R1 + 其他）** |
| **YMYL 表现** | 较弱 | 较弱 | 强 | 中 | 中（公众号医疗 / 金融大号） |
| **B 端 / 协作场景** | 弱 | 弱 | 中（百度文库） | 中（钉钉） | **强（腾讯文档 + 会议 + 企业微信）** |
| **多模态** | 强（视频转写） | 弱 | 强（文心 5.0 全模态） | 中 | 中（视频号字幕） |

**这张五大厂对照表的研究价值**：

- 五个平台的**核心引用源完全不同**——这是仓库最重要的发现
- 如果实验验证差异显著 → "通用中文 GEO 策略"概念**彻底破产**，必须按平台分流
- **生态封闭程度**形成谱系：DeepSeek（无生态）→ 千问（弱）→ 豆包 / 文心（中强）→ **元宝（最封闭）**
- 元宝的"多模型路由"是中文场景独有的复杂度

## 反假说 / 怀疑信号

调研中观察到的不一致信号：

1. **"36 亿篇公众号文章"是历史规模，不等于实际 RAG 池**——大量历史文章可能因质量 / 政策 / 时效被过滤
2. **多模型路由透明度低**——用户可能误判某次回答的模型来源
3. **微信生态的反爬保护**——非腾讯模型确实抓不全，但腾讯模型自己的 RAG 配置也可能不开放全部历史文章
4. **春节红包后的"浅度用户"** —— 5734.6 万月活中实际深度使用比例可能远低，影响"被引用"的曝光意义

## 关于"接入方分歧"问题（G 类研究边界）

通过腾讯云大模型 API 接入的产品：

- 接入方使用混元 / R1 等模型
- 但**没有微信公众号 RAG 加成**——只是模型本身
- 因此"通过 API 接入元宝模型" ≠ "在元宝 APP 上被引用"

**含义**：与 DeepSeek / 千问类似——"被元宝引用"必须先界定具体入口。

## 在做实验前能下的结论

**可以下的（强）**：

- 元宝月活 5734.6 万（一手数据，国内 AI 应用第四）
- 元宝采用**多模型混合架构**（混元 + R1 + 其他）—— 这是中文场景独有的事实
- 元宝**与微信生态深度绑定**（公众号 / 视频号 / 文档 / 会议）—— 是结构性事实
- 微信公众号的封闭性使元宝在该信源上有**结构性垄断优势** —— 这是其他大厂无法跟进的护城河

**不能下的**：

- "公众号占元宝引用 X%"等量化数字
- 多模型路由的具体策略
- 微信内嵌 AI 搜是否完全封闭
- A/B/C/D/E/F/G 七个入口的具体差异程度
