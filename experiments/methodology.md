# 中文 GEO/SEO 实验方法学（v1）

> **状态**：方法学文档，**confidence: high**——这不是关于某个平台机制的假说，而是关于"如何严谨地做中文 GEO/SEO 实验"的规范。
>
> 所有 `experiments/` 下的具体实验**必须遵循本文档的最小要求**。新方法 / 新约束沉淀到 v2、v3。

## 一、实验类型分类

仓库内的实验分三类，每类有不同的方法学要求：

### 1. 观察性实验（observational）

**定义**：不干预内容、不发布新内容，只观察平台对**已有 query 集**的回答。

**例子**：

- 在豆包 APP 跑 100 个 query，统计引用源分布
- 同一 query 在 5 个平台跑，对比引用差异
- 同一 query 每天跑 1 次，连跑 30 天，看时间序列变化

**优点**：

- 成本低、速度快
- 不需要发布内容、不需要等收录
- 可大规模并行

**局限**：

- 只能观察"现状"，不能证明"因果"——A 平台引用 B 内容多，不一定意味着 B 内容的某个特征导致了引用
- 受平台用户画像、时段、地理位置影响

**适用场景**：仓库现阶段（缺账号 / 缺生产力工具）的主要实验类型。

### 2. 对照实验（A/B）

**定义**：发布两个或多个内容变体（控制其他变量），观察**新发布的变体之间**的可见性差异。

**例子**：

- 同主题、同长度、同发布平台，A 组无引用 / B 组引用央媒，看 N 个 query 下被豆包引用的概率
- 同关键词、不同结构（FAQ vs 散文）的小红书笔记，看搜索排名差异

**要求**：

- 控制变量必须明确列出（什么变了 / 什么没变）
- 必须有最小样本量（建议每组 ≥ 3 个内容点）
- 必须等待收录窗口（2-4 周）
- 必须区分"被收录"和"被引用"——这是两个不同信号

**适用场景**：仓库进入"一手研究"阶段后的核心实验类型。

### 3. 干预实验（intervention）

**定义**：对**已有内容**做修改（增加引用 / 改写结构 / 改时间戳），观察前后差异。

**要求**：

- 必须有"做改动前"的基线数据
- 必须考虑平台缓存、检索更新延迟
- 不能改变和实验目标无关的内容（避免污染信号）

**风险**：

- 修改已发布内容可能影响真实业务
- 平台可能识别"频繁修改"为可疑信号

**适用场景**：高优先级、有真实账号资源的实验。

## 二、必须的实验设计要素

任何 `experiments/<name>.md` 必须包含：

### 1. 假设（hypothesis）

明确陈述："如果做 X，会观察到 Y"。**不能写"探索 X"或"研究 X"**——含糊的假设导致结论也含糊。

✅ 好的假设：
- "豆包对头条内容的引用比例显著高于第三方权威媒体"
- "在中文 LLM 中，引用央媒的内容被引用率比无引用的内容高 ≥ 20 %"

❌ 不合格的假设：
- "研究豆包的引用机制"
- "看看引用源分布是怎样的"

### 2. 自变量、因变量、控制变量

| 类别 | 含义 | 例 |
|------|------|----|
| 自变量 | 你刻意改变 / 选择的维度 | 平台（豆包 vs DeepSeek）、内容版本（A vs B）、query 类型 |
| 因变量 | 你测量的指标 | 被引用次数、引用源排名、引用准确率 |
| 控制变量 | 你尽量保持不变的变量 | 时间窗口、查询语言、采集账号、地理位置 |
| **混杂变量** | 你**知道但控制不了** | 平台个性化推送、用户历史、模型 A/B 测试 |

**特别强调"混杂变量"**——中文 GEO 实验中常见的混杂：

- **平台个性化**：豆包 / 千问可能基于账号历史给出不同答案
- **A/B 测试**：平台自身在跑实验，用户落入哪一桶不可知
- **联网检索的实时性**：同 query 不同时刻可能引用不同实时内容
- **多模型路由**（元宝特有）：同 query 可能被路由到不同模型

**应对方式**：

- 用**全新账号 / 匿名账号**降低个性化影响
- 跑**多次重复**（建议每 query ≥ 5 次）平均掉随机性
- 在**不同时段**采样（早 / 中 / 晚 / 周末）
- 记录**采集时刻的快照**——平台在变，半年后无法复现

### 3. 样本量（sample size）

最小要求：

| 实验类型 | 最小 query 数 | 最小重复次数 / query |
|----------|----------------|------------------------|
| 观察性（单平台） | 30 | 5 |
| 观察性（跨平台） | 30 × 平台数 | 3 |
| 对照（A/B） | 每组 ≥ 3 内容点 + 30 query 测试 | 3 |
| 干预 | 至少 5 个内容点 + 时间序列采样 | 每天 1 次连续 14 天 |

> 这是**最小要求**，不是最优。可信度评估会同时看样本量和方法严谨度。

### 4. 数据采集协议

每个实验必须明确：

- **采集账号**：账号 ID、账号年龄、是否会员、设备型号、操作系统、应用版本
- **采集时间**：精确到分钟（平台行为有时段性）
- **采集地点**：IP 地理位置（影响 LBS 类 query）
- **采集方式**：手动 / 浏览器自动化（Playwright / Puppeteer）/ 抓包 / API
- **联网状态**：是否启用搜索 / 联网（豆包 / 文心等都有"联网搜索"开关）
- **历史清空**：是否清空对话历史 / 是否新会话
- **平台版本**：APP / Web / API（同模型可能不同入口表现不同——见各平台 citation-mechanism 卡片）

### 5. 评分 rubric（关键）

"被引用" 不是单一信号，至少要分维度：

| 维度 | 说明 | 评分范围 |
|------|------|----------|
| **是否被提及** | 答案文本中是否出现品牌 / 内容名 | 0 / 1 |
| **是否被作为引用源** | 答案下方的"引用源"卡片中是否出现 | 0 / 1 |
| **引用源排名** | 在引用源列表中的位置（1, 2, 3...） | 1-N |
| **引用准确性** | 答案对引用源的转述是否准确 | 0-2（错 / 部分 / 准确） |
| **引用上下文** | 引用是正面 / 中性 / 负面 | -1 / 0 / 1 |
| **引用关联度** | 引用源与 query 的相关度 | 0-2 |

**默认主指标**：

- 观察性实验：**引用源出现频次**（在 N 次重复中被引用的比例）
- 对照实验：**引用源出现频次差** + **引用源排名差**
- 干预实验：**引用源出现频次的前后变化**

### 6. 可复现性级别

每个实验在 frontmatter 中标注 reproducibility（low / medium / high）：

| 级别 | 标准 |
|------|------|
| **high** | 数据集 + 脚本 + 环境 + query 集都公开，第三方一键复跑 |
| **medium** | 方法描述完整，但需要平台权限 / 账号才能复跑 |
| **low** | 仅描述结论，无原始数据 / 脚本 |

仓库**最低要求 medium**——low 级别的实验不收录。

## 三、反偏差检查清单

写完实验设计后，过一遍这份清单：

### 选择偏差（Selection Bias）

- [ ] query 集是怎么挑的？是否覆盖不同类型 / 行业 / 难度 / 长尾？
- [ ] 平台账号是新建还是用了主用户账号？（个性化污染）
- [ ] 时段是否覆盖工作日 / 周末 / 早晚？

### 确认偏差（Confirmation Bias）

- [ ] 假设是预先写好的，还是看了数据才"凑出来"的？
- [ ] 是否有可能 falsify 假设的反向 query？
- [ ] 数据分析时是否盲化（不知道哪条来自哪个变体）？

### 测量偏差（Measurement Bias）

- [ ] 评分 rubric 是预先定的，还是看了答案才定的？
- [ ] 不同评分人之间一致性如何？（如果可能，用 ≥ 2 个评分人 + Cohen's Kappa）
- [ ] 是否区分了"被提及" vs "被引用为来源"？

### 报告偏差（Reporting Bias）

- [ ] 失败 / 不显著的结果也要报告，**不能只挑符合假设的写**
- [ ] 是否报告样本量和置信区间？
- [ ] 是否避免使用"显著提升""大幅领先"这类未量化的 GEO 服务商话术？

### GEO 服务商话术的反向校验

特别警惕这几类话术：

- "提升 N 倍 / N%"——问：基线是什么？样本量？
- "AI 引用率达 X%"——问：哪个 AI？哪类 query？测算方式？
- "权威信源被采信率 92%"——问：定义？方法？
- "内容收录时间从 N 天缩短至 M 小时"——问：内容池规模？是否对照组？

**仓库实验在结果展示时主动避免这些话术**——用具体数字、置信区间、p 值（如有）、原始观察次数。

## 四、与英文学术研究的关系

借鉴：

| 论文 | 借鉴点 |
|------|--------|
| **Princeton GEO**（KDD 2024） | 9 个内容策略作为"假说起点"，实验设计参考其 query 集分类 |
| **C-SEO Bench**（NeurIPS 2025） | **多竞争方对抗设计**——如果可行，本仓库也应做"多内容方都用 GEO 时的衰减实验"；同时 C-SEO Bench 的"GEO 多数无效"是**反向锚点**，结果分析时主动 cross-check |
| **AutoGEO**（ICLR'26） | 偏好学习方法学（如能扩展到中文模型上） |
| **E-GEO**（电商 benchmark） | 电商 query 集设计（千问 / 豆包电商 query 实验复用） |
| **ConflictingQA** | LLM 对"相关性 vs 学术语调"的偏好提示——警告"权威语调"可能不如想象中重要 |

**不直接复用**：

- 英文 query 集（语种 + 文化 + 用户行为完全不同）
- 平台列表（ChatGPT / Perplexity / Gemini ≠ 中文五大厂）
- 信源分类（Wikipedia / Reddit / Quora ≠ 微信公众号 / 百家号 / 知乎）

## 五、中文 GEO 实验的特殊挑战

英文研究里不存在或不严重的问题：

### 1. 多入口 / 多模型路由

每个中文大厂都有多入口（详见各 citation-mechanism 卡片）。元宝甚至有多模型路由。**研究"被 X 引用"必须先界定哪个入口**。

### 2. 移动端为主 + 反爬保护

- 豆包 / 千问 / 元宝主要是 APP，Web 端可能限速 / 限制功能
- 浏览器自动化抓 APP 比抓 Web 难得多（需要 Appium / 模拟器 / 硬件控制）
- 平台对自动化检测严格（高频请求被风控）

**应对**：

- 优先用 Web 端做实验
- 接受小样本但高质量（每个数据点人工跑 + 截图存档）
- 使用 [Claude in Chrome / Playwright MCP] 等工具半自动化（详见 BACKLOG `meta-experiment-tooling`）

### 3. 账号风控

- 新账号在某些平台触发"风险用户"标签
- 多个测试账号被关联识别后行为可能受限

**应对**：

- 单账号多设备 vs 多账号单设备的取舍——倾向后者，但接受小样本
- 接受真实数据 ≠ 完整数据的现实

### 4. 联网搜索的 RAG 实时性

不同时刻同 query 在同平台可能引用不同的"刚发布"内容——这不是模型行为，是检索增强的实时性。

**应对**：

- 区分"模型本身的偏好" vs "实时检索的影响"
- 必要时关闭联网搜索做对照（如 UI 提供该开关）
- 在分析时把"近 X 天发布"的内容单独标记

### 5. 多模态信源（中文场景特别强）

- 豆包高度依赖抖音视频转写
- 元宝有视频号字幕
- 文心 5.0 原生全模态

**应对**：研究多模态 query 时区分"文本引用源" vs "多模态引用源"。

## 六、数据存储与脱敏

### 存储

- **小数据（<1MB）**：仓库内 `experiments/<name>/data/` 子目录（仅在必要时使用，目录深度第三层）
- **中等数据（1-100MB）**：外部存储（Gist / Google Drive / S3）+ 仓库内放校验和与链接
- **大数据 / 隐私敏感**：外部存储 + 仅给指定贡献者访问权限

### 脱敏

- 实验账号 ID **必须脱敏**——只保留账号年龄、是否会员、设备型号等元数据
- 真实业务数据脱敏到行业 + 量级
- 如果用了真实品牌做对照，**只在数据公开时披露**——避免让品牌方反向操作

## 七、实验前的最小检查清单

写实验设计文档时，自检：

- [ ] 假设是预先写好的？
- [ ] 自变量、因变量、控制变量都明确列出？
- [ ] 样本量符合最小要求？
- [ ] 评分 rubric 是预先定义的？
- [ ] 反偏差清单走过一遍？
- [ ] 数据采集协议详细到能复现？
- [ ] 可复现性级别已标注？
- [ ] 风险（账号风控 / 业务影响 / 法律合规）评估过？
