# 中文 GEO/SEO 实验 query 集（v1）

> **状态**：query 池 v1，覆盖 8 个核心 query 类型 × 每类 10-15 个 query。本集为**初始基线**，后续实验跑过后会沉淀 v2、v3。
>
> **设计原则**：
> 1. 中文原生 query（不是英文翻译）—— 避免文化偏差
> 2. 覆盖不同**意图类型**和**难度梯度**
> 3. 包含 **YMYL** 和 **创意 / 闲聊**等极端类型作为对照
> 4. 每个 query 标注**设计意图**——为什么选它、能测什么
> 5. **可扩展**——v1 之后按 query 类型补充，保持向后兼容

## 一、为什么需要分层 query 集

不同 query 类型下，平台的引用机制差异巨大（详见各 platform/citation-mechanism 卡片）：

| Query 类型 | 触发的平台行为 |
|-----------|------------------|
| 资讯（"今天发生了什么"） | 时效性 RAG、新闻信源 |
| 决策（"哪个品牌好"） | 评测 / 商品 / 排行榜信源 |
| 知识（"X 是什么"） | 百科 / 学术信源 |
| YMYL（医疗 / 金融 / 法律） | 权威机构 / 专业站信源 |
| 创意（"写一首诗"） | 模型自身知识，无 RAG 或弱 RAG |
| 技术（"如何实现 X"） | 技术社区 / 文档信源 |
| 电商（"推荐 X 商品"） | 电商生态 / 商品库 |
| 本地（"附近的 X"） | LBS / 本地生活信源 |

**实验设计时不能跨类型混合**——把"今天的新闻"和"如何做菜"放一起会让效应混杂。

## 二、Query 集分类

每类 10-15 个 query，按"难度 / 复杂度"梯度排列。所有 query 设计意图见每条注释。

测试**时效性偏好**和**新闻信源结构**。

| ID | Query | 设计意图 |

| info-06 | 苹果今年发布会的主要新品 | 时效 + 品牌 + 决策类边缘 |

### 4. YMYL 类（医疗 / 金融 / 法律）

测试**权威信源加权**和**E-E-A-T 中文版**。

⚠️ **重要警告**：YMYL 类 query **不能用真实医疗 / 法律建议来评估实验**——只用于测试"是否引用了权威信源 / 是否有合规警示"等元层面信号。

| ID | Query | 设计意图 |
|---|---|---|
| ymyl-01 | 高血压患者饮食上要注意什么 | 医疗 - 健康饮食（边缘 YMYL） |
| ymyl-02 | 糖尿病早期症状有哪些 | 医疗 - 疾病知识 |
| ymyl-03 | 信用卡逾期会有什么影响 | 金融 - 个人征信 |
| ymyl-04 | 个人所得税专项附加扣除有哪些 | 金融 - 税务（政策类） |
| ymyl-05 | 劳动合同到期不续签有什么补偿 | 法律 - 劳动法 |
| ymyl-06 | 房贷利率怎么计算 | 金融 - 房贷 |
| ymyl-07 | 离婚财产怎么分割 | 法律 - 婚姻 |
| ymyl-08 | 心绞痛和心肌梗死的区别 | 医疗 - 鉴别诊断（专业性高） |
| ymyl-09 | 基金定投适合普通人吗 | 金融 - 理财（决策类边缘） |
| ymyl-10 | 工伤认定的标准是什么 | 法律 - 工伤 |

> 这一类 query 的实验**额外要求**：
> - 必须评估"答案是否包含合规警示"（"建议咨询医生 / 律师 / 专业人士"等）
> - 不能基于实验答案做任何真实决策
> - 实验结果展示时去除任何可被误用为医疗 / 法律建议的内容

## 三、跨类型对照对（特殊设计）

某些 query 是**对**出现的，专门测某个变量：

### 对照对 1：同主题 vs 信源加权

- `know-05` "什么是 RAG（检索增强生成）" vs `tech-07` "RAG 的基本架构是怎样的"
- 设计意图：同主题、不同提问方式（概念 vs 实操），观察是否触发不同信源（百科 vs 技术社区）

### 对照对 2：同 LBS 商业类 vs 不同地域

- `local-01` 北京朝阳健身房 vs `local-03` 杭州本地餐厅
- 设计意图：同 LBS 类型、不同城市，观察平台对地方信源的处理

### 对照对 3：通用 vs YMYL

- `dec-08` 适合敏感肌的面霜推荐 vs `ymyl-01` 高血压饮食注意
- 设计意图：同样是"推荐 / 建议"类，但 YMYL 触发权威信源加权

### 对照对 4：决策 vs 电商

- `dec-05` 想买跑鞋有什么推荐 vs `comm-01` 推荐一双白色运动鞋 200 元以内
- 设计意图：同品类、不同明确度（开放 vs 价格 + 颜色明确），观察是否触发"AI 下单"路径（千问）

### 对照对 5：英文 vs 中文版（对照英文学术研究）

- `dec-10` 适合长跑的轻量缓震跑鞋（中文）→ 对应 E-GEO 论文中"running shoes for marathon"
- 设计意图：将英文 GEO benchmark 的部分 query 中文化，做交叉验证

## 四、Query 选择的偏差检查

写完 query 集后，回看：

- [ ] 是否覆盖**男性 / 女性偏好**的 query？（避免性别偏差）
- [ ] 是否覆盖**一线 / 三四线**的 LBS query？（避免地域偏差）
- [ ] 是否覆盖**高净值 / 低净值**消费的 query？
- [ ] 是否包含 query **明确知道答案** 的（事实类）和**没有标准答案** 的（创意类）？
- [ ] 是否避免选择"对自己假设有利"的 query？

## 五、版本管理

- **v1（2026-05-01）**：初版，80 个 query × 8 个类型
- **v2 计划**：根据 v1 实验结果增补；预计加入"长尾 query"（5+ 词）和"多模态 query"（含图片描述）
- 每个新 query 走 PR，注释设计意图

## 六、使用指引

- 单个实验**不必跑完全部 query**——根据实验目标选 1-2 个类型即可
- **跨平台对照实验**应至少跨 3 个 query 类型
- 同 query 在不同实验中的结果可以**累积比较**（前提是采集协议一致）
- 引用本 query 集的实验在 frontmatter 中写 `query_set: query-set-v1`，并列出实际使用的 query ID
